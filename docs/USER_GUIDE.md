# 洛书完整使用教程

本文面向直接下载安装洛书的用户，说明模块安装、App 安装、字体导入、单字体应用、组合字体、字重选择、恢复系统字体和故障排查。

## 1. 下载正确的文件

请只从仓库的 [Latest Release](https://github.com/xgl34222220-ops/LuoShu/releases/latest) 下载正式版。

正式 Release 包含：

```text
LuoShu-<版本>.zip              模块包，内置正式 App
LuoShu-<版本>.zip.sha256       模块校验文件
LuoShu-App-<版本>.apk          独立正式 App
LuoShu-App-<版本>.apk.sha256   App 校验文件
```

模块包和独立 APK 中的 App 使用同一把固定签名。不要把旧 `.debug` App 与正式 App 混装；Android 因签名不同拒绝覆盖时，应先卸载旧 Debug App，再安装正式 APK。

## 2. 安装模块

1. 打开 Magisk、KernelSU、SukiSU Ultra、APatch 或 Mountify。
2. 选择 `LuoShu-<版本>.zip` 安装。
3. 等待刷写完成。
4. 完整重启设备。

刷写阶段会部署文件并在模块覆盖生效前扫描本机原厂字体配置，但不会在 Recovery 或安装脚本中生成大型复合字体。

## 3. 安装或更新 App

模块包始终内置原生洛书 App，有两种安装方式：

- 在 Root 管理器中点击洛书模块的“操作”按钮；
- 直接安装 Release 中的 `LuoShu-App-<版本>.apk`。

刷写环境能够调用 Android 包管理器时，模块会尝试自动覆盖更新 App；无法调用时，会在首次完整开机后补装一次。自动安装失败时，使用模块“操作”按钮重试。

## 4. 准备字体

洛书支持常见 TTF、OTF、TTC、TrueType `glyf`、CFF、CFF2 和可变字体。

推荐方式：在 App 中使用系统文件选择器导入字体。

也可以手动复制到：

```text
/sdcard/LuoShu/fonts/
```

字体模块 ZIP 可放入：

```text
/sdcard/LuoShu/import/
```

洛书只提取其中的字体文件，不执行第三方模块脚本。

### 字体角色要求

中文基底：

- 包含常用中文；
- 包含英文字母和数字；
- 包含常用标点。

英文字体：

- 包含 `A–Z`、`a–z`；
- 包含常用拉丁标点。

数字字体：

- 包含 `0–9`；
- 包含常用数字标点。

图标字体、彩色字体、损坏字体、伪装扩展名和缺少关键字符的字体会被拦截。

## 5. 扫描字体库

1. 打开洛书 App。
2. 进入“字体库”。
3. 点击扫描或刷新。
4. 等待字体名称、格式、字重、可变轴和覆盖能力分析完成。

首次扫描字体较多时会比后续扫描慢。字体内容和修改时间没有变化时，洛书会复用缓存结果。

## 6. 直接应用单个字体

只想把系统字体整体替换为一套字体时：

1. 在中文、英文和数字三个槽位选择同一个字体；
2. 选择相同的标准字重；
3. 点击生成并应用。

满足快速应用条件时，洛书会优先使用单字体路径，不进行不必要的多次大型合成。

## 7. 生成组合字体

需要中文、英文和数字使用不同字体时：

1. 在“组合”页选择中文字体；
2. 选择英文字体；
3. 选择数字字体；
4. 分别选择需要的字重或可变轴；
5. 检查预览；
6. 点击“生成并应用复合字体”；
7. 等待任务达到 100%；
8. 完整重启设备。

组合字体以中文字体作为完整基底，只把英文和数字负责的字符写入基底，因此不会依赖系统缺字回退来拼接三套字体。

## 8. 字重与可变字体

### 静态字体

- 只显示字体文件真实存在的字重；
- 多字重静态字体默认只生成当前选择的字重；
- 不会因为检测到多个字重，就无提示生成 100–900 全部档位；
- 切换到其他字重时，再生成对应的当前字重。

### 可变字体

- 读取真实 `wght`、`wdth`、`opsz`、`slnt` 等设计轴；
- 轴范围来自字体本身，不使用伪造范围；
- 真正包含 `wght` 轴的字体可以使用完整可变字重能力。

## 9. 为什么需要重启

Android 的字体缓存、应用进程和系统服务不会同时安全热刷新。洛书不会通过只重启 SystemUI 来宣称字体已经完整生效。

生成并应用完成后，请完整重启设备。覆盖升级模块后，首次启动保持系统默认字体，需要重新应用一次字体或字体组合。

同一次开机内建议只提交一次系统文字字体变更，避免字体缓存和挂载状态不一致。

## 10. 设备自适应原厂字体清单

洛书不是通过手机品牌选择一套固定文件名清单。安装模块时，它会先扫描**当前这台手机**的真实字体结构，再生成设备独立的清单。

扫描范围包括常见的：

```text
/system
/system_ext
/product
/my_product
/vendor
/odm
/oem
/my_region
/hw_product
```

扫描器会：

- 查找实际存在的 TTF、OTF 和 TTC；
- 解析各分区中的 `fonts.xml`、`font_fallback.xml`、`fonts_customization.xml` 等配置；
- 记录系统 UI 字体的真实路径、所属分区、XML 来源、family、alias、字重、样式与 TTC index；
- 读取 UPEM、`hhea` 和 `OS/2` 字体度量；
- 识别主题字体覆盖和已有字体挂载，避免把主题字体误当成原厂字体；
- 根据系统指纹决定复用清单还是在 OTA 后重新扫描。

### 应用字体时的顺序

1. **优先读取当前设备清单**；
2. 按清单中发现的真实 UI 字体路径建立 systemless 覆盖；
3. 每个写入槽位再次校验字体文件；
4. 只有清单缺失、过期、损坏或没有发现可映射槽位时，才进入厂商静态规则或最小 AOSP 回退。

所以，ColorOS 和 HyperOS 不是洛书的使用白名单。其他品牌、其他机型和其他 ROM 同样会先扫描本机，然后按照本机清单直接应用字体。

当前运行时可直接映射的主要字体分区是：

```text
/system/fonts
/system_ext/fonts
/product/fonts
/my_product/fonts
/vendor/fonts
```

`/odm`、`/oem`、`/my_region`、`/hw_product` 会参与诊断、去重和 ROM 特征识别。若某个厂商把核心 UI 字体完全放在私有 framework、主题服务、加密资源或其他非标准动态加载机制中，清单可能只能发现部分字体槽位；这种情况需要提供真机诊断报告来补充专用增强适配。

不同 ROM 可以显示相同的“可替换 UI 槽位数”，但这只是数量相同，不代表使用同一份固定模板。实际路径、分区、XML 来源和字体度量仍是每台设备独立生成。

## 11. 恢复系统字体

在 App 中选择“恢复系统字体”后：

1. 等待恢复任务完成；
2. 完整重启设备；
3. 确认系统字体恢复正常。

恢复失败时不要连续反复提交任务。先进入日志页查看失败阶段，再导出诊断报告。

## 12. 卸载模块

1. 先在 App 中恢复系统字体；
2. 完整重启；
3. 在 Root 管理器中卸载洛书模块；
4. 再次完整重启；
5. 需要时单独卸载洛书 App。

卸载脚本会清理洛书创建的动态挂载和状态文件，但不会删除用户放在 `/sdcard/LuoShu/fonts/` 中的字体。

## 13. 常见问题

### 我的 ROM 不在测试列表，能不能用

可以尝试，并且不会直接套用 ColorOS 或 HyperOS 的固定清单。安装时仍会执行完整本机扫描，应用时优先使用扫描结果。

“未在测试矩阵中”只表示维护者还没有完成该设备的整套真机回归，不等于模块拒绝运行。安装后请先查看刷写日志中的原厂字体文件数、可替换 UI 槽位数和 ROM 识别结果，再使用一套授权字体进行测试。

如果日志显示已经“按设备原厂清单覆盖 N 个真实字体槽位”，说明当前应用走的是本机自适应清单，而不是通用 AOSP 回退。

### 组合任务很慢

大型 CJK 字体、CFF/CFF2、TTC 或复杂可变字体需要更多处理时间。静态多字重字体只组合当前所选字重；相同组合再次生成时会优先使用缓存。

任务长时间没有阶段变化时，到日志页查看是否卡在字体解析、角色检查、复合、验证或映射阶段。

### 显示“字体组合失败”

常见原因：

- 字体文件损坏；
- 中文基底缺少必要字符；
- 英文或数字字体缺少关键字形；
- 扩展名和真实格式不一致；
- 存储空间或可用内存不足；
- 后台任务被系统强制终止；
- 其他字体模块同时覆盖相同路径。

不要只截取最终“失败”卡片。请同时导出日志和诊断报告。

### 应用里部分文字没有变化

以下内容通常不经过 Android 系统字体映射：

- 应用自带字体；
- 输入法键帽；
- 图片、贴纸和路径化文字；
- 游戏或阅读器的私有字体引擎；
- 网页 CSS 下载字体；
- 部分 WebView 内容。

这是无 Hook 系统字体模块的功能边界。

### 更新模块后字体恢复默认

这是安全升级行为。新版本首次启动不会自动复用未经当前版本重新验证的旧负载。打开 App 重新应用一次字体或组合即可。

### App 无法覆盖安装

正式 App 与旧 Debug App 签名不同。卸载包名带 `.debug` 的旧测试 App，再安装 Release 中的正式 APK。字体和模块状态保存在模块与共享目录中，卸载 Debug App 不会直接删除字体文件。

## 14. 导出诊断报告

出现问题时：

1. 打开“日志”；
2. 刷新任务状态；
3. 导出诊断报告；
4. 检查并删除可能包含的私人路径、文件名或设备信息；
5. 在 Issue 中附上脱敏报告。

提交问题需要提供：

```text
设备型号：
ROM / Android 版本：
Root 管理器及版本：
挂载环境：
洛书版本：
字体格式和体积：
中文 / 英文 / 数字选择：
刷写日志中的原厂字体文件数：
刷写日志中的可替换 UI 槽位数：
日志是否显示按设备原厂清单覆盖：
复现步骤：
实际结果：
预期结果：
```

请勿公开上传没有授权的字体文件。

## 15. 机制支持与真机验证

### 机制支持

洛书会在不同品牌和 ROM 上执行同一套设备清单扫描流程，不依赖预置机型名单。能在受支持分区和配置中发现系统 UI 字体槽位的设备，可以直接使用扫描结果应用字体。

### 已完成或正在进行的重点回归

- ColorOS 16 / Android 16；
- HyperOS 3 / Android 16；
- Magisk；
- KernelSU；
- SukiSU Ultra；
- APatch；
- Mountify。

更完整的真机验证状态见 [TEST_MATRIX.md](TEST_MATRIX.md)。该表是验证进度，不是支持白名单。
